# Config Files Analysis - What Can/Cannot Be Moved

## ✅ Moved to `config/` Directory

### 1. `eslint.config.mjs` → `config/eslint.config.mjs`
**Can be moved:** Yes
**Why:** ESLint supports `--config` flag
**Changes made:**
- Updated `package.json` script: `"lint": "eslint --config config/eslint.config.mjs"`

### 2. `postcss.config.mjs` → `config/postcss.config.mjs`
**Can be moved:** Yes
**Why:** PostCSS auto-discovers config in parent directories
**Changes made:**
- None needed - PostCSS will find it automatically

### 3. `vitest.config.ts` → `config/vitest.config.ts` (already moved)
**Can be moved:** Yes
**Why:** Vitest supports `--config` flag
**Changes made:**
- Package.json scripts already updated

### 4. `playwright.config.ts` → `config/playwright.config.ts` (already moved)
**Can be moved:** Yes
**Why:** Playwright supports `--config` flag
**Changes made:**
- Package.json scripts already updated

## ❌ Cannot Be Moved (Must Stay in Root)

### 1. `package.json`
**Cannot be moved**
**Why:** npm/Node.js require this at the project root. This is the project manifest and defines the workspace boundary.

### 2. `next.config.ts`
**Cannot be moved**
**Why:** Next.js requires this file to be in the project root where `next` commands are executed. It's hardcoded to look for config at the root.

### 3. `tsconfig.json`
**Cannot be moved**
**Why:** TypeScript requires this at the project root. The TS compiler (`tsc`) looks for it starting from the root. Moving it would break:
- VS Code IntelliSense
- TypeScript compilation
- Path resolution (`@/` aliases)
- IDE integrations

### 4. `next-env.d.ts`
**Cannot be moved**
**Why:** This is auto-generated by Next.js and must be in the root. Next.js creates/updates it during build and expects it at the root for TypeScript type definitions.

### 5. `components.json`
**Cannot be moved (but could be)**
**Why:** This is for shadcn/ui components. While technically it could be moved with configuration, it's conventional to keep it at root. Most shadcn examples and tooling expect it at the root.
**Recommendation:** Leave at root for compatibility with shadcn CLI and documentation.

## Summary

**Movable to `config/`:**
- ✅ `eslint.config.mjs`
- ✅ `postcss.config.mjs`
- ✅ `vitest.config.ts`
- ✅ `playwright.config.ts`

**Must stay in root:**
- ❌ `package.json` (npm/Node requirement)
- ❌ `next.config.ts` (Next.js requirement)
- ❌ `tsconfig.json` (TypeScript requirement)
- ❌ `next-env.d.ts` (Next.js auto-generated)
- ❌ `components.json` (conventional, tooling expects it)

This is a good balance - we moved what we could while respecting the constraints of the build tools and ecosystem conventions.
